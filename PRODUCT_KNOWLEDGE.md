## 1. Product Summary

**Vizhva** is a SaaS portfolio builder that lets professionals create and publish a public portfolio page at `vizhva.me/<username>`. Built and operated by **GARI TECH**.

### What It Does

A user signs up, completes a 7-step onboarding wizard, and their portfolio is immediately live at a public URL. From the dashboard they can customize the design (10 templates, drag-and-drop section reordering, per-section colors and animations), add rich content across 13 section types, view real-time analytics, and optionally connect a custom domain.

### Problem Solved

Developers, designers, and freelancers need a professional online presence but don't want to maintain a handcrafted site. Vizhva offers a no-code, opinionated alternative that still produces high-quality output — not a generic résumé site but a styled, shareable portfolio with a public URL.

### Type of Product

SaaS, B2C. Web-only. Monetized via subscription tiers:
- **Free**: Full portfolio at `vizhva.me/<username>`, all 13 section types, 6 templates
- **Pro** ($9/mo or ₹499/mo INR): 4 additional templates, custom domain, full analytics breakdown
- **Teams** ($299/mo or ₹9,999/mo INR): pricing shown on landing page; backend and features not yet implemented

### Maturity Level

**Late MVP / early production.** Evidence:
- Full auth, billing (Dodo Payments), and analytics implemented end-to-end — not stubs
- 10 templates fully built and gated correctly
- Supabase migrations in sequence, covering all features
- Pytest suite covering analytics, domain, and profile routers
- Multiple deployment configs (Render, Koyeb, Railway, Docker, Netlify)
- Cold-start UX (4s "waking up" message) suggests free-tier hosting in production

### Backend

**`backend/main.py`**
- Does: Creates FastAPI app titled `"Vizhva API"`; configures CORS (comma-separated `FRONTEND_URL` + hardcoded localhost); creates `Limiter` instance; registers 7 routers; exposes `GET /health`
- Depends on: All 7 routers, `slowapi`, `fastapi.middleware.cors`
- Depended on by: Uvicorn (entry point)

**`backend/dependencies.py`**
- Does: Two singletons — `get_supabase_client()` (lazy init with `SUPABASE_SERVICE_KEY`) and `get_current_user()` (FastAPI dependency; verifies Clerk JWT via JWKS; auto-derives JWKS URL from `CLERK_PUBLISHABLE_KEY` by base64-decoding); returns `{sub, email, role}`
- Depends on: `supabase`, `jwt` (PyJWT), `httpx`
- Depended on by: Every router via `Depends(get_current_user)` and `get_supabase_client()`

**`backend/routers/profile.py`**
- Does: `/api/profile/*` — the most complex router; `GET /me/data` fetches profile + 13 tables (13 sequential queries); `PUT /me` updates profile; `PATCH /me/username` enforces `USERNAME_RE` regex; `GET /check-username` with suggestions; `POST /me/suggest-username` generates `firstname-lastname` slug; `POST /me/avatar` uploads to Supabase Storage `avatars` bucket; `DELETE /me` cascade-deletes everything; **`GET /{username}` MUST be last** — catches all unmatched paths
- Depends on: `dependencies`, `pydantic`, `re`, `uuid`
- Depended on by: `main.py`

**`backend/routers/sections.py`**
- Does: `/api/sections/*` — `PUT /order`, `PUT /visibility`, `PUT /settings`, `PUT /extras`; all 9 section type upserts (`experience`, `education`, `certifications`, `services`, `testimonials`, `books`, `publications`, `quotes`, `custom`); `_upsert()` helper: delete all for `profile_id`, re-insert
- Depends on: `dependencies`, `pydantic`, `uuid`
- Depended on by: `main.py`

**`backend/routers/billing.py`**
- Does: `/api/billing/*` — `POST /checkout` builds Dodo payload with `product_cart`, `customer`, `return_url`, `metadata.clerk_user_id`; `POST /portal` fetches portal URL from Dodo; `POST /webhook` verifies HMAC-SHA256, handles `subscription.active/renewed/on_hold/failed/cancelled`; `DODO_TEST_MODE=true` → `test.dodopayments.com`
- Depends on: `httpx`, `hmac`, `hashlib`, `json`, `dependencies`
- Depended on by: `main.py`

**`backend/routers/analytics.py`**
- Does: `/api/analytics/*` — `POST /view` is rate-limited 30/min, uses `BackgroundTasks` to insert `page_views` row; device detection from UA string; IP hashed with SHA-256; country from `CF-IPCountry` header; `GET /me` calls `supabase.rpc("get_portfolio_analytics", ...)`
- Depends on: `dependencies`, `slowapi` (local `Limiter` instance to avoid circular import), `hashlib`, `uuid`
- Depended on by: `main.py`

**`backend/routers/domain.py`**
- Does: `/api/domain/*` — `POST /connect` saves domain + sets `domain_verified=false`; `GET /verify` calls `_verify_cname()` which uses `dns.resolver` to walk CNAME chain and check for `VIZHVA_CNAME_TARGET = "vizhva.me"`; `DELETE /` removes domain
- Depends on: `dns.resolver` (dnspython), `dependencies`
- Depended on by: `main.py`

**`backend/routers/projects.py`**
- Does: `/api/projects/*` — standard CRUD (list, create, update, delete, reorder); `POST /{id}/image` uploads to Supabase Storage `project-images` bucket
- Depends on: `dependencies`, `pydantic`, `uuid`
- Depended on by: `main.py`

**`backend/routers/skills.py`**
- Does: `/api/skills` — `PUT /` upserts skill list (delete + insert); `/social` upserts social_links the same way
- Depends on: `dependencies`, `pydantic`, `uuid`
- Depended on by: `main.py`

## 6. Data Flow — Major User Journeys

### Journey 1: New User Sign-Up and Onboarding

```
1. User visits / → LandingPage → clicks "Get Started Free"
2. Redirect to /signup → Clerk <SignUp> component (hosted UI)
3. User completes sign-up (email or Google OAuth)
4. Clerk issues JWT (RS256, signed with Clerk private key)
5. Clerk redirects to /dashboard/editor (afterSignUpUrl in ClerkProvider)

6. DashboardLayout mounts:
   → setTokenGetter(getToken)  ← injects Clerk JWT getter into Axios interceptor
   → fetchProfile()            ← calls GET /api/profile/me/data
   → setTimeout(4000, () => setSlowLoad(true))  ← cold-start UX

7. Backend: GET /api/profile/me/data
   → get_current_user() verifies JWT via PyJWKClient (JWKS from Clerk)
   → get_supabase_client() returns singleton
   → SELECT profiles WHERE id = clerk_user_id
   → If no profile: auto-insert with generated username from email prefix
   → Return profile + 13 empty table results

8. EditorPage mounts:
   → Checks: !onboardingComplete (localStorage) && !hasProfileData
   → Shows <OnboardingFlow />

9. OnboardingFlow (7 steps, all local state until final step):
   Step 0: Template selection → local preview state
   Step 1: First + Last name entry
           → useEffect: debounce 400ms → POST /api/profile/me/suggest-username
           → Backend: _generate_username(first, last) → "firstname-lastname" slug
           → suggestedUsername shown as URL preview
   Step 2: Photo upload + Bio
   Step 3: Skills (from preset categories or custom input)
   Step 4: Projects (title, description, URL)
   Step 5: Social links (github, linkedin, twitter, website)
   Step 6: Username confirmation (can edit suggested; real-time availability check)

10. User clicks "Launch portfolio →":
    → PUT /api/profile/me   {template, full_name, last_name, tagline, bio, is_public: true}
    → POST /api/profile/me/avatar  (if photo uploaded)
    → PUT /api/skills       [{name, order_index}, ...]
    → POST /api/projects    (one per project with title)
    → PUT /api/skills/social [{platform, url}, ...]
    → PATCH /api/profile/me/username  (if username was changed from suggested)
    → fetchProfile()
    → onComplete() → editorStore.setOnboardingComplete(true) → persisted to localStorage

11. Portfolio is live at vizhva.me/<username>
    → "Your portfolio is live!" toast shown for 6s
```

### Journey 4: Pro Upgrade (Billing)

```
1. User clicks "Upgrade to Pro" (editor Pro modal or Settings billing section)
2. createCheckout() → POST /api/billing/checkout
   → Backend: fetch email + full_name from profiles
   → POST https://test.dodopayments.com/checkouts (or live)
     body: {
       product_cart: [{product_id: DODO_PRODUCT_ID, quantity: 1}],
       customer: {email, name},
       return_url: FRONTEND_URL + "/dashboard/settings?upgrade=success",
       metadata: {clerk_user_id: profile_id}
     }
   → Returns {checkout_url: "https://checkout.dodopayments.com/..."}
3. window.location.href = checkout_url  ← user leaves to Dodo checkout

4. User pays on Dodo → Dodo fires webhook to POST /api/billing/webhook
   → Verify HMAC-SHA256: hmac(DODO_WEBHOOK_SECRET, raw_body).hexdigest()
   → event_type == "subscription.active":
     → metadata.clerk_user_id → UPDATE profiles SET is_pro=true,
       dodo_subscription_id=..., subscription_status='active'

5. Dodo redirects user to /dashboard/settings?upgrade=success
   → SettingsPage useEffect detects ?upgrade=success
   → Shows "You're now on Pro!" banner
   → fetchProfile() → profile.is_pro now true
   → Pro templates unlocked; analytics Pro sections visible
```

### Journey 5: Custom Domain Setup (Pro)

```
1. Pro user goes to Settings → Custom Domain section
2. Enters "portfolio.johndoe.com" → clicks "Connect"
3. connectDomain("portfolio.johndoe.com") → POST /api/domain/connect
   → Normalize: strip https://, lowercase, take hostname only
   → UPDATE profiles SET custom_domain='portfolio.johndoe.com', domain_verified=false
   → Return {dns_instructions: {type: "CNAME", name: domain, value: "vizhva.me", ttl: 3600}}

4. UI shows DNS table: add CNAME record in DNS registrar

5. User adds CNAME, waits for propagation (up to 24h), clicks "Verify DNS"
6. verifyDomain() → GET /api/domain/verify
   → _verify_cname("portfolio.johndoe.com"):
     → dns.resolver.Resolver(lifetime=5s).resolve(domain, "CNAME")
     → Walk results: str(rdata.target).rstrip(".").lower()
     → If == "vizhva.me" or ends with ".vizhva.me" → True
     → NoAnswer or NXDOMAIN or Exception → False
   → If verified: UPDATE profiles SET domain_verified=true
   → Return {verified: true/false}

7. If verified: green "✓ Domain verified and active" shown
   NOTE: Vizhva verifies DNS ownership but does not serve traffic at the
   custom domain. Actual routing would require Netlify domain aliases or
   a reverse proxy — this is not implemented.
```

## 10. Product Intelligence

### Ideal Customer

- **Primary**: Software engineers and designers at the individual contributor level (IC) who want a professional portfolio without maintaining a personal website. Particularly engineers between their 2nd and 6th year who are job-seeking or freelancing.
- **Secondary**: Bootcamp graduates who need a portfolio immediately. They produce the highest volume of signups but lower Pro conversion.
- **Tertiary**: Academic researchers and content creators who want to showcase publications, talks, and media — the extended section types (books, publications, quotes) strongly signal this audience.
- **Not well-served yet**: Non-technical professionals (the terminal template and default aesthetic skew technical).

### Core Value Proposition

Speed to live (< 5 minutes) with professional output — the onboarding wizard and username auto-generation eliminate the "blank page" problem, and the templates are genuinely design-quality, not generic résumé builders.

### Notable Architectural Decisions and Rationale

**1. All DB writes through FastAPI (never direct from frontend)**
The service role key never leaves the server. Even though `@supabase/supabase-js` is installed in the frontend package, `supabaseClient.js` is a dead-code stub. RLS is a safety net, not the primary access control layer.

**2. Clerk over Supabase Auth**
Clerk provides hosted UI (sign-in/sign-up pages), social OAuth, organizations (future), and better DX. The JWKS-based JWT verification is stateless and works cleanly with a separate FastAPI backend. The JWKS URL auto-derivation from `CLERK_PUBLISHABLE_KEY` (base64 decode the suffix) saves one env var.

**3. `_upsert()` pattern (delete + re-insert)**
Simpler than diff-based updates for list sections where order matters. The `order_index` is set by position in the request array. Safe for sections with < 100 items. Would become a performance issue at scale.

**4. PostgreSQL RPC for analytics**
`get_portfolio_analytics()` runs all aggregations (zero-filled 30-day series, top-N countries, device breakdown) in a single SQL call. This is the right call — doing this in Python would require fetching all `page_views` rows into memory.

**5. Per-router `Limiter` in analytics.py**
The main app `Limiter` is registered on `app.state.limiter`. If `analytics.py` imported and reused that instance, it would create a circular import (routers imported from main, main imports from routers). The solution: create a local `Limiter` instance in `analytics.py` with the same key function.

**6. `BackgroundTasks` for view recording**
The view record doesn't block the HTTP response. If Supabase is slow, the visitor gets their portfolio in < 100ms regardless. Correct pattern.

**7. Local state in OnboardingFlow, saved all at final step**
No per-step API calls — all saves happen when "Launch portfolio →" is clicked. This avoids partial data states and simplifies the UX. The trade-off: if the user closes the browser mid-onboarding, no data is saved. Acceptable at this stage.

### Potential Weaknesses and Technical Risks

**1. `GET /api/profile/me/data` makes 13 sequential Supabase queries**
This is the most expensive and most frequently called endpoint (called on every dashboard load). At Supabase's latency, this is ~200-500ms. A single PostgreSQL function (like `get_portfolio_analytics`) would reduce this to one round trip.

**2. Cold-start on Render free tier**
The 4s "waking up" UX and 3-retry profileStore logic exist because the backend sleeps after inactivity. This is a significant UX gap for new users. A cron job to ping `/health` every 10 minutes would mitigate this cheaply.

**3. Custom domain routing is DNS-only, not traffic-routing**
Vizhva verifies CNAME ownership and stores `domain_verified=true`, but there is no mechanism to actually serve `portfolio.johndoe.com` — no Netlify domain alias, no reverse proxy, no wildcard subdomain routing. This feature is incomplete.

**4. No rate limiting on most endpoints**
Only `POST /api/analytics/view` is rate-limited (30/min). `GET /api/profile/{username}` (which triggers 13 DB queries), `POST /api/billing/checkout`, and all section upserts have no rate limiting. This is an abuse surface.

**5. `supabaseClient.js` in frontend is dead code**
It creates a Supabase JS client with the anon key. This client would be blocked by RLS but its existence could confuse future developers into thinking direct frontend DB access is intended.

**6. Pillow is installed but unused**
`pillow==11.0.0` is in `requirements.txt` but not imported anywhere in the current routers. Likely a remnant of a planned image-processing feature (resizing, compression). Adds ~20MB to the container image.

**7. Teams plan is marketing-only**
The $299/mo Teams tier is displayed on the landing page and in pricing, but there is no team management, shared portfolios, multi-user workspace, or Teams-specific checkout. This risks customer complaints.

### Natural Next Features

1. **PostgreSQL function for `/me/data`** — collapse 13 queries into one, exactly like analytics
2. **Keep-alive cron job** — ping `/health` every 10 min to prevent Render cold starts
3. **Custom domain traffic routing** — Netlify wildcard domain alias or Cloudflare Worker proxy
4. **Rate limiting on all endpoints** — add `@limiter.limit("60/minute")` to profile and section routes
5. **Teams workspace** — Clerk organizations map naturally; shared portfolios, admin dashboard
6. **AI bio/tagline suggestions** — Claude API; natural upsell for a portfolio builder audience
7. **Portfolio analytics email digest** — weekly "your portfolio had 42 views" email
8. **Template preview in settings** — screenshot thumbnails for template picker
9. **Dark mode** — CSS custom properties are already in place; just needs a second theme block

### Code Patterns

| Pattern | Definition |
|---------|-----------|
| **`_upsert(supabase, table, profile_id, rows)`** | Delete all rows for `profile_id` in `table`, then bulk-insert `rows`. Used for all list sections in `sections.py` and `skills.py`. |
| **`safe_single(query)`** | Appends `.limit(1).execute()` and returns `result.data[0]` or `None`. Avoids Supabase's `.single()` which raises on no-match. |
| **`BackgroundTasks`** | FastAPI feature: adds async work to run after the HTTP response is sent. Used for `page_views` insert in analytics. |
| **`setTokenGetter(fn)`** | Function exported from `api/api.js`; called by `DashboardLayout` with Clerk's `getToken` on mount; injects JWT into every subsequent Axios request via interceptor. |
| **`persist` middleware** | Zustand middleware that saves specified state keys to localStorage. `editorStore` persists only `onboardingComplete` under key `'vizhva-editor'`. |
| **3-retry fetch** | `profileStore.fetchProfile()` retries up to 3 times with 3s delay between attempts. Designed to survive Render cold-start timeouts. |
| **Pro gate** | In `EditorPage`: `if (tplMeta?.pro && !profile?.is_pro) { setShowProModal(true); return }`. Checks `is_pro` from DB before allowing template switch. |
| **`isINR` locale detection** | `navigator.language === 'en-IN' \|\| Intl.DateTimeFormat().resolvedOptions().timeZone === 'Asia/Kolkata'` — shows ₹499 pricing for Indian locale. |
| **`TEMPLATES` array** | Defined in both `EditorPage.jsx` and `OnboardingFlow.jsx`; 10 entries with `key`, `label`, `icon`, `pro` fields. |
| **`ALL_SECTIONS` array** | 14 sections defined in `EditorPage.jsx`: hero, about, skills, projects, experience, education, certifications, services, testimonials, books, publications, quotes, contact, custom. |
| **`VIZHVA_CNAME_TARGET`** | Constant `"vizhva.me"` in `domain.py`. The DNS verification checks if the CNAME chain resolves to this value. |
